## What This Project Is

A role-based internal HR management tool for small companies (10–20 employees). Covers employee lifecycle management, daily attendance tracking, and leave management. Built on Supabase (DB + Auth + Storage + Edge Functions) with a React frontend.

**v1 scope:** Employee Management + Attendance & Leave Tracking.
**Out of scope (v1):** Payroll, TDS/PF/ESI, biometric integration, shift swap requests.

## Key Conventions

### When to use Edge Functions vs direct Supabase client

| Operation | Use |
|---|---|
| Any read (SELECT) | Direct Supabase JS client — RLS handles access |
| Supabase Realtime subscriptions | Direct Supabase JS client |
| Any write with business logic | Edge Function (see `docs/EDGE_FUNCTIONS.md`) |
| Any action requiring a server-side timestamp | Edge Function |
| Any action that sends notifications/email | Edge Function |
| Simple reference data inserts (Owner adding a department) | Direct Supabase JS client is acceptable if no side effects |

### Supabase client usage
- **Browser client** (`src/lib/supabase.ts`): uses `VITE_SUPABASE_URL` + `VITE_SUPABASE_ANON_KEY`. Use for all frontend queries.
- **Service-role client** (`supabase/functions/_shared/supabase.ts`): uses `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY`. Use inside Edge Functions only, never exposed to the browser.

### Calling Edge Functions from the frontend
```typescript
const { data, error } = await supabase.functions.invoke('function-name', {
  body: { ... }
})
```
The user's JWT is automatically attached by the Supabase JS client.

### TanStack Query keys
Use the pattern `['domain', 'resource', id?]`:
```typescript
['employees', 'list']
['employees', 'detail', employeeId]
['attendance', 'records', { employeeId, month, year }]
['leave', 'applications', 'pending']
```

### Zod schemas
Define one Zod schema per form and per Edge Function request body. Co-locate with the feature. Export the inferred TypeScript type alongside:
```typescript
export const submitLeaveSchema = z.object({ ... })
export type SubmitLeavePayload = z.infer<typeof submitLeaveSchema>
```

### Error handling
- Edge Functions return `{ error: { code: string, message: string } }` on failure (see `docs/EDGE_FUNCTIONS.md` for codes).
- In React: display `error.message` in a toast. Do not show raw error objects to users.
- Never swallow errors silently. Always surface them in the UI.

### Date/time
- All dates stored in the DB are IST (UTC+5:30) calendar dates or UTC timestamptz.
- Use `date-fns` or `date-fns-tz` for all date manipulation. Never use raw `new Date()` arithmetic for IST conversion.
- The server (Edge Function) is the authority for "now". Never rely on `new Date()` on the client for anything that gets stored.

### Immutable columns
Never send these fields in UPDATE payloads — the server will reject them:
- `leave_applications`: `employee_id`, `leave_type_id`, `from_date`, `to_date`, `working_days_count`, `applied_at`
- `employee_lifecycle_events`: any column (INSERT only)
- `audit_logs`: any column (trigger-written only)

### Role-based rendering
Use the `useRole()` hook and `<RoleGuard>` component. Never hardcode role checks inline in JSX — it becomes impossible to audit.
```typescript
const { role } = useRole()
// or
<RoleGuard allow={['owner', 'hr']}><AdminPanel /></RoleGuard>
```

supabase functions serve

supabase functions deploy <function-name>

## Non-Negotiable Rules

1. **RLS is always on.** Never disable RLS on any table to make something work. Fix the policy instead.
2. **Edge Functions set timestamps.** `check_in_time`, `check_out_time`, `applied_at` — never sent from the client.
3. **No hard deletes.** Set `is_active = false`. Filter on `is_active = true` in all queries.
4. **`account_number_encrypted` is never returned to the frontend.** Strip it at the Edge Function layer.
5. **Business rules live in Edge Functions or DB constraints.** Never enforce them only in React.
6. **Presigned URLs only for document access.** Never expose raw storage paths to the frontend.
